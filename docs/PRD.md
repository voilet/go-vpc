## 1. 项目背景与业务目标

### 1.1 现状描述
目前全网部署约 50,000 台 x86 服务器，分布在以下复杂网络环境中：
* **公网/拨号：** 具备公网 IP 但地址随拨号频繁变动。
* **大局域网：** 位于多层 NAT 之后，无公网访问权限。
* **跨地域：** 节点遍布全球/全国各 IDC 或边缘机房。

### 1.2 核心目标
构建一个高性能、可扩展的 **SD-WAN** 虚拟内网，实现：
1.  **全网互联：** 5W 台设备在逻辑上处于同一大二层或三层网络（100.64.0.0/10）。
2.  **高性能传输：** 基于 WireGuard 内核加速，满足 x86 平台的高吞吐需求。
3.  **弱网自愈：** 针对拨号设备 IP 变动，实现秒级连接恢复。

---

## 2. 系统架构设计

系统采用 **控制面 (Control Plane)** 与 **数据面 (Data Plane)** 分离的 SDN 架构。



### 2.1 核心组件
1.  **Management (Go):** 管理中心。负责设备注册、身份校验、虚拟 IP (VIP) 分配及 ACL 策略存储。
2.  **Signal Server (Go/gRPC):** 异步信令服务。负责 Peer 之间的握手撮合，维护 5W 个长连接状态。
3.  **Relay/DERP (Go):** 中继节点。针对双对称 NAT 打洞失败的情况，提供基于 HTTP/UDP 的流量转发。
4.  **Agent (Go):** 客户端。封装 `wireguard-go`，负责本地路由表维护、NAT 探测及数据加密传输。

## 3. 功能需求

### 3.1 设备接入与识别
* **[F101] 自动身份注册：** Agent 首次启动生成 Ed25519 密钥对，向 Management 注册并获取唯一的隧道 IP。
* **[F102] 设备指纹：** 结合 CPU 序列号与网卡 MAC 生成唯一 ID，防止由于克隆镜像导致的 IP 冲突。

### 3.2 智能组网与 NAT 穿透
* **[F201] NAT 类型探测：** Agent 集成 STUN 协议，启动时上报当前 NAT 级别（全锥型、受限型、对称型）。
* **[F202] 动态信令握手：** 当 A 需访问 B 时，通过 Signal Server 交换双方的 Endpoint 信息，触发 UDP 打洞。
* **[F203] 拨号漫游 (Roaming)：** 拨号设备 IP 变化后，Agent 立即向中心同步，存量隧道通过 WireGuard 的心跳机制自动重定向，无需重新握手。

### 3.3 网络拓扑与策略
* **[F301] 按需 Mesh (On-demand Mesh)：** 不采用全互联拓扑。仅当节点间有流量往来时，才动态建立加密隧道，降低单机内存开销。
* **[F302] 访问控制策略 (ACL)：** 支持在 Management 层面配置策略（如：分组 `Web-Server` 只能单向访问 `DB-Cluster` 的 3306 端口）。

## 4. 技术指标 (Non-functional Requirements)

| 指标维度 | 详细要求 |
| :--- | :--- |
| **支持规模** | 系统设计容量需支撑 50,000+ 活跃节点 |
| **P2P 成功率** | 锥形 NAT 间穿透成功率 > 90% |
| **隧道延迟** | P2P 模式下额外时延 < 1ms; 中继模式下取决于 Relay 部署密度 |
| **单机资源** | Agent 常驻内存 < 64MB，CPU 闲时占用率 < 1% |
| **安全性** | 隧道流量 100% ChaCha20-Poly1305 加密；信令通过 TLS 传输 |

---

## 5. 关键业务流程

### 5.1 节点上线流程
1.  **Agent** 启动 -> 执行 NAT 探测 -> 向 **Management** 发起 Auth 申请。
2.  **Management** 分配隧道 IP -> 下发初始 Peers 列表。
3.  **Agent** 建立本地 `wg0` 网卡 -> 监听指定 UDP 端口。

### 5.2 复杂 NAT 下的连接建立
1.  **节点 A** 发送数据包至 **节点 B** 的隧道 IP。
2.  **Agent A** 发现无 B 的 Endpoint -> 向 **Signal** 订阅 B 的坐标。
3.  **Signal** 通知 **Agent B** 主动向 A 发送 UDP 探测包。
4.  若 P2P 失败，**Management** 指示双方连接至最近的 **Relay** 节点。

## 6. 运维与监控需求
* **[M1] 链路可视化：** 在后台展示 5W 节点的在线率、穿透方式（P2P/Relay）及流量分布。
* **[M2] 故障预警：** 当 Relay 节点 CPU 超过 80% 或中心信令队列堆积时触发告警。
* **[M3] 自动更新：** Agent 支持下发指令实现自升级，确保全网版本统一。

## 7. 后续规划
* **Phase 1:** 完成基于 `wireguard-go` 的核心 Agent 与中心 Management 的闭环，支持 1000 节点测试。
